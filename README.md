# tavern-chronicler

**给 SillyTavern 换一个会记账的大脑。**

一个接在 [SillyTavern](https://github.com/SillyTavern/SillyTavern) 和 LLM 之间的桥接服务：ST 继续负责它擅长的前端渲染与角色卡生态，回复生成与**长对话记忆管理**交给桥——每轮回复后，一个后台记忆任务用文件工具维护结构化的战役档案（状态账本、编年史、NPC 台账、伏笔索引），替代"向量化检索"这种在长篇叙事后期必然崩坏的模糊召回。

**双通道**：有 Claude 订阅的走 [Claude Agent SDK](https://docs.claude.com/en/api/agent-sdk)（复用 Claude Code 登录态，零 API 费用）；没有的填一个任意 **OpenAI 兼容端点的 API key**（GPT / Grok / 中转均可），记忆系统全套照用。

```
SillyTavern（前端渲染 / 角色卡 / 预设 / 世界书）
   │  OpenAI 兼容 API
   ▼
tavern-chronicler（本服务）
   ├── 战役识别：聊天键（ST 扩展随请求盖章）优先、对话指纹回退（同卡多聊天各自隔离）
   ├── 定向回溯（可选）：小模型出检索词 → 进程内毫秒级扫归档＋BM25 → 精准注入旧细节
   ├── 回复路径（同步、快）：注入战役档案 → sdk（Claude 订阅）或 api（自备 key）生成 → SSE 流式返回
   ├── 记忆路径（异步、不阻塞）：后台任务读写该战役目录下的 Markdown 档案（sdk / agent / api 三种后端）
   └── 管理通道（ws://…/admin）：ST 扩展面板实时改配置、看日志、查用量
```

## 为什么不用向量化记忆

跑团/长篇 RP 需要的是**状态一致性**，而向量检索给的是**语义相似度**——这是两类问题：

- "背包里有什么"是账本问题，不是检索问题
- "三十楼前埋的伏笔"在嵌入空间里没有"剧情重要性"这个维度
- 线性时间与因果链在向量库里完全丢失

Agent 维护的结构化档案（下表）在每轮回复前作为权威状态注入，早期对话原文可以放心截断。

## 环境要求

- Node.js ≥ 20
- 一个能连自定义 OpenAI 兼容后端的 SillyTavern（1.12+）
- 二选一：
  - **sdk 通道（默认）**：已安装并登录 [Claude Code](https://claude.com/claude-code)（本服务通过 Agent SDK 复用其登录态，**无需单独的 API key**；订阅额度即可驱动）
  - **api 通道**：任意 OpenAI 兼容端点的 API key（OpenAI / xAI / 中转站均可），无需安装 Claude Code

> sdk 通道**不需要开着 Claude Code**。Agent SDK 会自行拉起运行时子进程；只要凭据在磁盘上（登录过一次），保持运行的只有本服务和 SillyTavern。

### 无 Claude 订阅（GPT / Grok 用户）

三处配置切到 api（面板改或环境变量均可），其余功能——战役档案、定向回溯、BM25、导入补课、掷骰（熵池）、管理面板——完全一致：

| 配置 | 值 |
|---|---|
| `CHAT_MODE=api` + `CHAT_API_URL` / `CHAT_API_KEY` | 回复走你的端点；模型名默认跟随 ST 里填的 |
| `MEMORY_MODE=agent` + `MEMORY_API_URL` / `KEY` / `MODEL` | 记忆用函数调用 agent 增量编辑档案（需模型支持工具调用，GPT-5 / Grok-4 级前沿模型适用）；模型偏弱或中转不转发工具调用时改 `MEMORY_MODE=api`（全文件重写协议，任何模型可用） |
| `RECALL_MODE=api` + `RECALL_API_URL` / `KEY` / `MODEL` | 回溯出词接个便宜快的模型即可，延迟只有 1-2 秒 |

此形态下 `@anthropic-ai/claude-agent-sdk` 是可选依赖，安装失败不影响运行。

api 通道对反代/中转站的常见不规范形态做了容错：先确认上游可读才向 ST 回 200（上游报错时 ST 能看到真实错误码与原因，429 透传、超时 504、其余上游错误 502）；SSE 按事件边界解析（CRLF、多行 `data:`、无换行尾块均可），声明流式却返回整段 JSON 的上游会被自动转回 SSE，`[DONE]` 后不断连的上游会被主动收工；不支持 `stream_options` 的端点自动降级重试；推理模型的思维增量（`reasoning_content` 等）不透传但计入活跃度，只有思维没有正文时报错而非静默空回复；`finish_reason` 原样透传（`length` 截断不再伪装成正常结束）；所有上游请求带超时与客户端断开取消（ST 停止生成即取消上游，不再白烧 token）。

## 快速开始

```sh
git clone https://github.com/<you>/tavern-chronicler
cd tavern-chronicler
npm install
node server.mjs
# → st-claude-bridge listening on http://127.0.0.1:9377/v1
```

Windows 下也可以直接双击仓库里的 `start.bat`（SillyTavern 用它自带的 `Start.bat` 启动）。

然后在 SillyTavern 里：

1. **API 连接**（插头图标）→ API 选 **Chat Completion**，来源选 **Custom (OpenAI-compatible)**
2. 端点填 `http://127.0.0.1:9377/v1`，API Key 随便填（本服务不校验）
3. 连接后选择模型，开卡即用——预设、世界书、正则、前端脚本全部照常工作

## 管理面板（SillyTavern 扩展）

配套的 ST UI 扩展放在本仓库的 [`st-extension` 分支](https://github.com/afu6609/tavern-chronicler/tree/st-extension)（独立分支、根目录即 manifest，桥代码与插件代码互不混杂），在酒馆的扩展设置抽屉里提供 **Tavern Chronicler** 面板：

- **配置热改**：下表除 `PORT` / `MEMORY_ROOT` 外的所有配置项按组渲染成表单，改完即时生效（下一轮请求即用新值）、无需重启桥；修改持久化到 `memory/bridge-config.json`
- **旧对话导入 + 补课**：打开一个对话，点"导入当前对话"即可把全部楼层归档进战役档案（已有匹配战役则补全合并，timeline 轮号自动平移修正）；随后可选择"补课"——让记忆 agent 分批通读旧楼层构建编年史与状态账本，进度实时显示、中断可续跑。导入完成后回溯即刻可用（关键词检索直接扫原文），补课完成后轮号指针与梗概索引同样就位
- **战役管理**："定位战役"用当前对话的指纹找到对应档案（一个对话文件绑定一份记忆），管理卡片上可**编辑**五个档案 md（面板内直接改，白名单限定）、**重建档案**（原文件备份为 `*.pre-rebuild.bak` 后从头补课重新生成）、**删除战役**（软删除，移入 `memory/trash/` 可手动恢复）
- **日志实时流**：桥的控制台日志按 `[chat]` / `[recall]` / `[memory]` / `[dice]` 着色流入面板（连接时先回放最近 400 行），掷骰虚构告警等 ⚠ 行高亮——不用再盯桥的控制台窗口
- **缓存命中率**：面板顶部实时显示三条路径的累计平均命中率；每条日志的用量尾缀也带单次命中率
- **用量快照**：一键查看本次运行的分路径 token 汇总

面板分**设置 / 战役 / 日志**三个页签，配置组可折叠（记住展开状态），小屏下表单自动上下排布——手机上也能舒服调参、看日志。

安装（任选其一，装完刷新酒馆页面即可）：

- **ST 扩展安装器**：扩展 → 安装扩展，仓库地址填本仓库、分支填 `st-extension`
- **手动克隆**：`git clone -b st-extension https://github.com/afu6609/tavern-chronicler SillyTavern/public/scripts/extensions/third-party/tavern-chronicler`

面板默认连 `ws://127.0.0.1:9377/admin`，桥端口有改的话在面板顶部改地址。管理通道只监听本机且校验来源页面，外部网页无法连入；API 密钥类字段在面板侧始终打码。

## 配置

优先级：**`memory/bridge-config.json`（面板修改自动写入）> 环境变量 > 内置默认**。环境变量适合当作部署基线，日常调参用面板。键名三处一致：

| 变量 | 默认 | 说明 |
|---|---|---|
| `PORT` | `9377` | 监听端口（重启生效） |
| `CHAT_MODE` | `sdk` | 回复通道：`sdk`（Claude Code 订阅登录态）/ `api`（任意 OpenAI 兼容端点，无 Claude 订阅可用） |
| `CHAT_API_URL` | — | api 通道端点（如 `https://api.openai.com/v1`、`https://api.x.ai/v1`），配合 `CHAT_API_KEY`、`CHAT_API_MODEL`（留空则跟随 ST 请求里的模型名） |
| `BRIDGE_MODEL` | `claude-sonnet-5` | sdk 通道的默认回复模型（ST 里未选择或选择无效时的回退） |
| `CHAT_EFFORT` | SDK 默认（`high`） | 回复模型的推理力度：`low` / `medium` / `high` / `xhigh`。调低可显著减少思考 token、加快出字，代价是叙事规划变浅 |
| `RECALL_EFFORT` | `low` | 回溯出词模型的推理力度。提关键词是机械任务，默认 low 省时省 token |
| `RECALL_THINKING_BUDGET` | `2000` | 出词模型思考 token 上限（sdk 模式）。自适应思考在回忆密集的轮次会长考数千 token 拖过超时线；出词要果断不要深刻。设 `0` 恢复不设限 |
| `MEMORY_EFFORT` | `medium` | 记忆 agent 的推理力度（sdk 模式）。medium 比 high 显著缩短更新耗时，档案质量实测无损；设空字符串可回到 SDK 默认（high） |
| `MEMORY_MODEL` | 同 `BRIDGE_MODEL` | 记忆更新用的模型（sdk 模式），可设便宜档位省额度 |
| `MEMORY_MODE` | `sdk` | 记忆后端：`sdk`（订阅 agent 带文件工具增量编辑）/ `agent`（自配端点的函数调用 agent，同样增量编辑）/ `api`（自配端点全文件重写，兼容性最强） |
| `MEMORY_API_URL` | — | agent/api 模式端点（如 `https://xxx/v1`），配合 `MEMORY_API_KEY`、`MEMORY_API_MODEL` |
| `MEMORY_MAX_TURNS` | `30` | 记忆任务（sdk/agent 模式）的最大工具回合数，长回复大档案可适当调高 |
| `CATCHUP_BATCH` | `15` | 导入旧对话后"补课"时每批喂给记忆 agent 的轮数；批越大总批次越少、单批越慢 |
| `MEMORY_ROOT` | `./memory` | 战役档案根目录（重启生效） |
| `RECENT_TURNS` | `25` | 正文携带的最近对话轮数；更早的历史由战役档案接管 |
| `RECENT_TURNS_MAX` | `0`（关闭） | 窗口锚定模式：设为大于 `RECENT_TURNS` 的值后，窗口起点固定、正文纯追加（前缀缓存跨轮存活），涨到该上限才一次性收缩回 `RECENT_TURNS`。仅在两轮间隔小于缓存寿命（5 分钟）的快节奏对话中有收益，慢节奏对话请保持关闭 |
| `CONTINUE_PROMPT` | 中性续写指令 | 拼在 prompt 末尾的续写指令。默认不指定任何身份/视角（多角色、多人称、旁白式预设均不受干预），仅要求衔接上文自然续写；需要强制某种视角时可自行覆盖 |
| `RECALL_MODE` | `sdk` | 定向回溯后端：`sdk`（走 Claude Code 订阅）/ `api`（自配 OpenAI 兼容端点）/ `off` |
| `RECALL_MODEL` | `claude-haiku-4-5-20251001` | sdk 模式用的模型 |
| `RECALL_API_URL` | — | api 模式的端点（如 `https://xxx/v1`），配合 `RECALL_API_KEY`、`RECALL_API_MODEL` |
| `RECALL_BUDGET` | `6000` | 回溯注入内容的字符预算 |
| `RECALL_BM25` | `on` | 中文 1/2-gram BM25 打分层：命中超预算时按相关度取舍、字面失配时模糊兜底（纯本地计数，零额外调用）。`off` 退回纯字面匹配 |
| `RECALL_TIMEOUT` | `120000` | 回溯整体超时（ms），超时直接放弃、不阻塞回复。命中多时要出词+压缩两次调用，调小会掐掉恰恰最有价值的回溯 |
| `DICE_MODE` | `tool` | 掷骰机制：`tool`（进程内 MCP 工具，真随机决定成败）/ `pool`（熵池注入，零延迟）/ `off` |
| `DICE_TRIGGER` | `auto` | `auto` = 扫描预设/卡中的规则关键词，非跑团场景不触发；`always` = 总是启用 |
| `DICE_MAX_TURNS` | `12` | tool 模式下回复 agent 的工具回合上限，足够覆盖多次检定的战斗轮 |

### 可选模型

`/v1/models` 暴露全系列模型（全名），在 ST 的模型下拉框里直接选，逐条请求透传给 SDK：

- **Fable**：`claude-fable-5`
- **Opus**：`claude-opus-4-8` / `4-7` / `4-6` / `4-5` / `4-1` / `4-0`
- **Sonnet**：`claude-sonnet-5` / `4-6` / `4-5` / `4-0`

> 实际可用性取决于你的订阅档位；订阅不包含的模型（如 Fable 通常需要更高档位）请求时会报错，换一个即可。

## 战役档案（memory/campaigns/<战役>/）

**每个聊天自动对应一个独立战役目录**，不需要任何配置。识别分两级：

1. **聊天键（优先）**：装了 ST 扩展面板时，扩展会把当前聊天文件标识（`角色卡::聊天文件名`，群聊为群 id）随每个发往桥的请求盖章送达（经 `custom_include_body`，只对指向本桥的请求盖章、不外发），桥据此把战役和聊天文件**一对一绑定**——切换预设、正则改写历史等改变内容形态的操作都不再影响归属。已有战役会在下一次生成/导入时自动认领聊天键；聊天文件改名/分支后指纹回退会接住并维持匹配。
2. **对话指纹（回退）**：把每一轮消息哈希后与已存档战役比重合度，命中含生成回复在内的多轮即续接，否则开新档。没装扩展、或其他前端接入时全靠它，行为与从前一致。

**压缩形态防呆**：若请求中出现超长单条消息（>6 万字符，典型为 clewd/squash 类预设把全部历史压成一条），桥判定轮次结构不可信，本请求不关联任何战役——正常转发生成，但不归档、不记忆、不回溯，并在日志告警，避免预设脚手架污染真实档案。

每个战役目录里，由记忆管理器在每轮回复后维护的 Markdown 档案（**随时可以手动编辑**，agent 会尊重你的修改）：

| 文件 | 内容 |
|---|---|
| `world_state.md` | 游戏内时间 / 地点 / 天气 / 当前任务与目标 |
| `party.md` | 队伍 HP、状态、装备、金钱账本 |
| `npc_ledger.md` | 出场 NPC 的态度、承诺、已知信息 |
| `timeline.md` | 按事件压缩的编年史（只追加），每条事件标注来源轮号（#N），供定向回溯按号调取原文 |
| `foreshadowing.md` | 未回收的伏笔与悬念 |

另有三个由桥自动维护的归档（不要手动编辑）：

| 文件 | 内容 |
|---|---|
| `transcript.jsonl` | 完整对话原文（每行一轮），记忆 agent 可 Grep 查证旧细节，为后续"定向回溯"打底 |
| `discarded.jsonl` | 被重roll/编辑覆盖掉的旧回复，无损留档 |
| `meta.json` | 战役元数据（创建时间、标题等） |

回复前 `.md` 档案作为 `<campaign_memory>` 注入正文（transcript）之后——每轮易变的内容统一后置，保住 system+transcript 长前缀的提示词缓存；每个战役的记忆更新任务同一时间只跑一个，重叠时自动跳过（下一轮补记）。

**滚动备份**：记忆任务每次动笔前（常规更新 / 补课每批 / 面板手工保存），现有 `.md` 档案会先快照到 `memory/backups/<战役id>/<时间戳>-<来源>/`，环形保留最近 10 份——agent 万一改坏文件，直接拷回上一份即可；删除战役时备份随之清理。

记忆维护有三种后端（`MEMORY_MODE`）：

- **`sdk`**（默认）：Agent SDK 拉起带文件工具的 agent（走 Claude 订阅），在战役目录内增量编辑，还能 Grep `transcript.jsonl` 查证旧细节
- **`agent`**：自配 OpenAI 兼容端点的**函数调用 agent**——四个受围栏的工具（读档案/整写/精确替换/检索归档），与 sdk 模式同样做增量编辑；档案现文随任务一次性给出，通常 1-3 个工具回合完成。需要模型可靠支持工具调用（GPT-5 / Grok-4 级前沿模型适用；部分中转站对工具调用的转发是坏的，遇到就退 `api`）
- **`api`**：自配端点的"全文件重写"协议——档案现文 + 本轮交互一次性输入，模型按 `===FILE: 文件名===` 格式输出需更新文件的全文，桥校验白名单后落盘。不依赖工具调用能力，任何 OpenAI 兼容模型都能用；代价是没有归档查证能力，且过滤"变量/状态栏等非叙事内容"的质量取决于所配模型

两种模式的指令都明确要求：聊天原文中的变量标记、状态栏代码、思维链推演等非叙事内容不得写入档案，只提炼叙事事实。

**重roll 修正**：你在 ST 里点"重新生成"时，桥会对比归档检测出被弃用的回复（判定依据：请求停在待重新生成的用户消息上、存档多出尾巴），并在下一次记忆更新时要求 agent 修订档案中已不成立的内容——档案不会被"平行世界"的废稿污染。同位置内容变化（手动编辑消息、或 ST 正则脚本改写 assistant 消息后回传）则静默以 ST 版本为准，不触发修正。

> 从旧版（单战役布局）升级：启动时根目录的 `.md` 会自动迁入 `campaigns/legacy-*/`，并由升级后**第一个进行中的对话**（超过 2 轮）自动认领。所以升级后请先继续玩原战役一轮，再开新聊天。

## 定向回溯：用精确检索替代向量召回

长对话里"那个商人当初承诺过什么"这类问题，向量检索会给你几十条语义相近但张冠李戴的片段，外加一份常驻上下文的巨型剧情总结——token 消耗巨大且不精确。本项目的做法：

1. **出检索词**（一次小模型单轮调用）：输入只有编年史 + 最新消息（约 1-2k token），输出 1-4 个具体关键词和/或轮号指针，或判定"本轮无需检索"
2. **本地检索**（零 token、毫秒级）：进程内正则扫描 `transcript.jsonl` 中提示词窗口之外的早期轮次，命中轮附带前后文，去重、按预算截断
3. **按需压缩**：命中内容超过阈值才再花一次调用压成 ≤300 字备忘，随后以 `<archive_recall>` 注入正文之后（与 md 档案同在缓存友好的尾部区）

**轮号指针（关键词失效时的兜底）**：逐字匹配依赖"原文里有锚点词"（人名/地名/物品名），但有些内容天然无锚（"他当时是不是很不情愿"）。为此记忆 agent 写 timeline 时会在每条事件末尾标注来源轮号（`（#12-13）`，压缩合并时保留合并后的范围），出词模型可以输出 `"turns":["12","30-35"]` 直接按号调取原文——**梗概层当语义索引，轮号当指针解引用**，全程零嵌入。单个范围最多展开 20 轮，越界（落入提示词窗口内）的轮号自动忽略。

**BM25 模糊+排序层（`RECALL_BM25=on`，默认开）**：字面匹配之上再叠一层中文 1/2-gram BM25 打分——把每轮原文拆成"逐个汉字 + 相邻双字"的碎片（拉丁词整词保留），经典 Okapi 公式按词频饱和、稀有度加权（IDF 自动压低"的/了"这类常见字）、长文惩罚三要素打分。作用有二：①**排序**——命中超出 `RECALL_BUDGET` 时按相关度取舍，而不是按轮号先到先得；②**模糊兜底**——出词给"星陨匕首"、原文写的是"星陨石匕首"，字面匹配落空后靠双字碎片重叠仍能召回（最多补 6 轮、须过全库最高分 30% 的相对门槛并至少命中一个双字/整词，杜绝单字噪音）。纯本地计数、零额外调用，几百轮归档打分只要几十毫秒。日志会区分命中来源（`字面/邻轮/指针/模糊`）；管理通道另有 `recallProbe` 调试消息，可跳过出词模型直接验证给定关键词的命中与分数。

慢的从来不是文本扫描（几 MB 归档毫秒级扫完），而是 LLM 的多回合工具往返——所以这里没有开放式 agent 循环，全程最多两次单轮小调用。判定无需检索的轮次只花 ~1k token；需要检索的轮次注入量通常在几百字符量级（实测：50 轮归档中找回一个旧暗号，仅注入 192 字符）。

回溯只在归档长度超过 `RECENT_TURNS` 后才启动（窗口内的内容本来就在 prompt 里）。两种后端延迟差异明显：`sdk` 模式受子进程冷启动拖累（实测约 20s），**追求响应速度建议配 `api` 模式接轻量快速模型**（实测检索部分仅几十 ms，加上模型延迟通常 1-2s）。

## 真随机掷骰

LLM 自己"掷骰"永远偏向剧情需要——爽点前必出高点。本项目用两种机制根治：

- **`tool`（默认）**：进程内 MCP 工具。模型叙事到检定点时暂停、调用 `roll("1d20+3")` 拿到 `crypto` 级真随机结果，**再基于结果续写成败**——低点数会真实地改变剧情走向。代价是每次掷骰增加一个模型回合（几秒停顿）。
- **`pool`**：请求前预掷一批真随机数注入正文尾部，指示模型按序消耗。零延迟，但约束靠模型自觉，适合轻检定的叙事场合。

**非跑团场景零介入**：`DICE_TRIGGER=auto`（默认）下，桥扫描预设+卡+世界书中的规则关键词（检定 / d20 / 先攻 / 豁免 / DC / TRPG…），没有骰点语境的卡完全不启用、prompt 零污染。每次掷骰都打日志（`[dice] 1d20+3 = [3]+3 = 6`），可与回复正文对账。

## 用量统计

三条路径（回复 / 记忆 / 回溯）的 token 消耗全程统计：

- 每条日志尾部带 `[tokens: in X, out Y, cache rA/wB]`（SDK 调用会区分缓存读写）
- `GET /stats` 返回本次运行的分路径汇总与按战役累计（战役累计持久化在各自 `meta.json`）
- `/v1/chat/completions` 的响应（含流式末块）回填真实 `usage`，ST 端如有 token 显示可直接用

读数说明：`in` 是**未走缓存的新输入**（大预设场景下常见个位数，属正常——整个预设/卡/世界书都在 `cache w` 里首建、后续轮走 `cache r` 复读，缓存读的成本远低于全新输入）；`out` 含模型思考 token，会大于可见回复的换算量。带缓存的日志尾缀还有单次**命中率**（= 缓存读 ÷ 提示词总量），管理面板顶部实时显示三条路径的累计平均命中率——命中率低通常意味着两轮间隔超过缓存寿命（5 分钟）或中途换过模型，而非排布问题。

## 已知限制（MVP）

- **记忆有一轮延迟**：本轮的变化在下一轮才可见
- 多轮消息被展平为 transcript 交给 SDK（Agent SDK 是单提示模型），对格式极敏感的预设可能有细微差异
- 战役识别的极端边界：两个聊天若开场白与第一条玩家消息**完全相同**且都停留在第一轮，会被并档（一旦各自有了生成回复即可正确分流）
